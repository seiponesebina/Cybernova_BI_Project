1. System Overview
===============================================================================

CyberNova Analytics Ltd is a fictitious Gaborone-based AI cybersecurity start-up
serving SMEs, financial institutions, and government agencies across Southern
Africa. The CyberNova BI system analyses synthetic IIS web server logs to show
how potential customers interact with CyberNova's online products, especially
the flagship AI Cyber Assistant.

The system is implemented as a Streamlit Business Intelligence portal. It covers
the full BI workflow:

1. Configure data generation.
2. Generate synthetic IIS-style web logs.
3. Enrich logs with countries, services, sessions, campaigns, lead signals, and
   behavioural segments.
4. Validate the generated dataset.
5. Load the dataset into Streamlit.
6. Apply filters and role-specific analytics.
7. Present dashboards for Sales, Marketing, and Executive Management.
8. Export evidence and reports.
9. Deploy through GitHub and Streamlit.

All data is synthetic. The system does not use real CyberNova customers, real
server logs, real personal information, or live geolocation services.

4. High-Level Architecture
===============================================================================

The system follows this architecture:

config.yaml
  Defines random seed, date range, country weights, service definitions,
  campaign windows, anomaly rules, and output file paths.

generate_logs.py
  Generates synthetic IIS-style logs and enriches them with analytical fields.

data/output/
  Stores generated raw logs, enriched logs, Excel workbook, and generation
  summary.

validate_dataset.py
  Checks dataset completeness, record volume, distribution, status codes,
  anomalies, and generation quality.

cybernovaapp.py
  Main Streamlit application. Loads data, caches dataframes, applies filters,
  manages authentication, combines historical data with live replay rows, and
  renders dashboard views.

sales_views.py, marketing_views.py, executive_views.py
  Role-focused dashboard modules.

exports.py
  Export helpers for CSV, Excel, PDF, and summary reporting.

live_feed.py and data/live_stream_queue.csv
  Simulated live stream support for operational replay.

requirements.txt
  Dependency file used for local setup and Streamlit deployment.

8. ETL Pipeline
===============================================================================

The ETL process transforms generated web logs into dashboard-ready BI data.

Extract:

- Read enriched CSV from data/output/cybernova_enriched_logs.csv.
- Use raw CSV and Excel workbook as supporting artefacts.
- Read live replay rows from data/live_stream_queue.csv when live mode is used.

Transform:

- Parse timestamp and date fields.
- Cast boolean fields such as is_bot, is_warm_lead, has_demo_request, and
  has_ai_interest.
- Normalize live replay column names where required.
- Derive missing live fields such as service_name, status_class, hour, and
  engagement flags.
- Sort rows chronologically.
- Apply country, service, segment, date, and HTTP status filters.
- Exclude bot traffic by default when human-market analysis is required.

Load:

- Load filtered dataframes into dashboard views.
- Cache data using Streamlit caching.
- Use pickle/parquet cache files when available in cybernovaapp.py.
- Combine historical data and released live rows for replay dashboards.

Quality controls:

- Fixed random seed for reproducibility.
- Validation script for generated outputs.
- Generation summary by day.
- Invalid status flag support.
- Data freshness status support.
- Anomaly labels and recommended action fields.

9. Behavioural Segmentation
===============================================================================

The system classifies visitors into four behavioural tiers:

High-intent:

- Visitors interacting with demo, event, contact, or conversion-oriented pages.
- Used by Sales for follow-up and prioritisation.

Product-curious:

- Visitors interacting with product and service pages, especially AI and
  cybersecurity pages.
- Used by Marketing and Sales to understand active service interest.

General browser:

- Visitors using general website pages such as home, about, or lower-intent
  browsing paths.
- Used to measure awareness and top-of-funnel traffic.

Bot:

- Crawler or automated traffic detected through generation logic and user-agent
  behaviour.
- Excluded from most human-market analytics unless the user includes filtered
  traffic.

12. Live Feed Simulation
===============================================================================

The live feed is a synthetic replay, not a production server listener.

The replay reads rows from data/live_stream_queue.csv, releases a controlled
number of rows per refresh, and updates timestamps to simulate current activity.
cybernovaapp.py supports replay speeds:

- Slow.
- Normal.
- Fast.

The app can refresh the live view through streamlit-autorefresh. If the refresh
package is unavailable, the rest of the dashboard still works.

This design is appropriate because the project constraints state that real
CyberNova server logs are unavailable.

14. Security and Ethics
===============================================================================

Ethical controls:

- All data is synthetic.
- No real customer records are used.
- No real server logs are used.
- No real personal data is used.
- No live geolocation API is used.
- The project is safe for academic demonstration.

Prototype security:

- Role-based login is implemented for demonstration.
- Passwords are stored in source code, which is acceptable only for prototype
  demonstration.

Production security requirements:

- Store credentials as Streamlit secrets or environment variables.
- Hash passwords.
- Add proper user management.
- Add audit logging.
- Restrict data exports by role.
- Use HTTPS-only deployment.
- Review large file and data retention policies.

16. Known Limitations
===============================================================================

- Live monitoring is simulated through replay.
- Role authentication is not production-grade.
- Forecasting is simple and interpretive.
- Some generated data files are large and may require careful GitHub/Streamlit
  management.
- The dashboard is designed for BI demonstration rather than enterprise
  multi-user operations.

17. Maintenance Guide
===============================================================================

Monthly refresh process:

1. Update config.yaml if campaign windows, services, or date ranges change.
2. Run python generate_logs.py.
3. Run python validate_dataset.py.
4. Launch streamlit run cybernovaapp.py.
5. Review Sales, Marketing, and Executive dashboards.
6. Export evidence and reports.
7. Commit updated source and approved data artefacts to GitHub.
8. Confirm Streamlit deployment still loads.

Change management:

- Keep config changes separate from code changes where possible.
- Do not commit temporary log files unless needed as evidence.
- Avoid committing secrets.
- Keep requirements.txt updated when dependencies change.
- Retest Streamlit deployment after any dependency or file path change.
