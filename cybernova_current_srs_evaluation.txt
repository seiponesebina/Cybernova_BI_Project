SECTION 1 - LANGUAGES, FRAMEWORKS, AND LIBRARIES
--------------------------------------------------------------------------------

Primary language: Python

Python is the core implementation language for the CyberNova BI Portal. It is
used for synthetic IIS log generation, ETL, enrichment, behavioural
segmentation, dashboard logic, forecasting, exports, and local execution.

Frontend/UI languages: HTML and CSS inside Streamlit

The app uses Streamlit as the runtime framework, with substantial custom HTML
and CSS for dashboard layout, login design, branded cards, filter drawers,
status chips, and role-specific visual styling.

Configuration/data formats: YAML, CSV, Excel, Pickle

- config.yaml controls generation settings such as random seed, country
  distribution, campaign dates, service weights, and output paths.
- CSV stores raw IIS logs, enriched logs, and generation summaries.
- Excel workbooks are generated for richer decision-engine outputs.
- Pickle is used as a fast dashboard cache through
  cybernova_enriched_logs.fast.pkl.

Main framework: Streamlit

Streamlit powers the web app, page configuration, login flow, tab navigation,
global filters, cached data loading, auto-refreshing live fragments, dashboard
rendering, and local deployment.

Analytics and visualisation stack: Pandas, NumPy, Plotly

- pandas performs data loading, filtering, grouping, aggregation, enrichment,
  and export preparation.
- numpy supports numeric calculations, simulation, forecasting, random
  generation, and simple statistical operations.
- plotly renders the dashboard charts: bar charts, line charts, scatter plots,
  funnel charts, donut charts, gauges, maps, and forecast visuals.

Reporting/export stack: ReportLab, OpenPyXL, Kaleido

- ReportLab builds PDF reports.
- OpenPyXL supports Excel output through pandas.
- Kaleido is listed for Plotly image export support.

Synthetic data stack: Faker, PyYAML

- Faker supports realistic synthetic values where required.
- PyYAML reads the project configuration from config.yaml.

SECTION 3 - FUNCTIONAL REQUIREMENTS (FR1-FR12)
--------------------------------------------------------------------------------

FR1 - Generate >=1,000-row synthetic IIS W3C log dataset
Status: FULLY MET

Evidence:
- The enriched dataset contains 60,551 rows, far above the 1,000-row minimum.
- Raw IIS fields are present: date, time, IP address, method, URI, status code,
  user agent, bytes transferred, and response time.
- config.yaml defines Southern African country weights, with Botswana set at
  32%.
- generate_logs.py includes realistic peak-hour clustering, bot ratios,
  campaign spikes, anomaly days, 404 errors, and repeat-session logic.

No gaps.

--------------------------------------------------------------------------------

FR2 - Ingest and preprocess log file; parse fields; resolve IPs; handle malformed
      records; load a clean DataFrame; report records processed/cleaned/dropped
Status: MOSTLY MET

Evidence:
- cybernovaapp.py loads the enriched CSV or fast pickle cache through load_data().
- Timestamps and dates are parsed with pandas.
- Boolean BI columns are normalised for dashboard use.
- Country resolution is performed during generation using static mappings, which
  matches the SRS constraint that no live geolocation API is required.
- The app falls back gracefully if data cannot be loaded.

Gap:
- The SRS specifically requires reporting records processed, cleaned, and
  dropped. The current dashboard shows filtered row context but does not clearly
  expose a formal ETL summary such as "Loaded X rows; Y malformed records
  dropped." This is a small but explicit SRS gap.

FR5 - Simulate a real-time log feed that refreshes dashboard metrics without

Status: MET IN SPIRIT

Evidence:

SECTION 6 - WHERE THE CURRENT APP EXCEEDS THE SPECIFICATION
--------------------------------------------------------------------------------

1. Role-based login and dashboard access
   The app provides role selection and password-controlled access for Sales,
   Marketing, and Executive users. This was not required in the SRS but makes
   the prototype feel closer to a real client-facing BI portal.

2. Modular dashboard architecture
   The current version separates the application into role-specific modules:
   sales_views.py, marketing_views.py, executive_views.py, and exports.py.
   This improves maintainability and directly strengthens NFR5.

3. Large generated dataset
   The enriched dataset contains 60,551 records, exceeding both the FR1 minimum
   and the NFR6 50,000-row scalability target.

4. Export functionality
   PDF, CSV, and Excel-oriented export support goes beyond the SRS baseline and
   gives the system a stronger BI delivery layer.

5. Forecasting and target comparison
   Sales and Executive views include forecast-style charts, confidence bands,
   target lines, and business interpretation notes.

6. Branded UI and business-facing design
   The interface is more developed than a basic academic dashboard, with custom
   styling, role-specific density, live status indicators, and polished
   dashboard sections.

SECTION 7 - WHERE WORK REMAINS
--------------------------------------------------------------------------------

PRIORITY 1 - FR12 descriptive statistics and correlation

Add a visible statistics table showing mean, median, standard deviation, and
totals by country, service, and/or time period. Add an hour x service heatmap or
correlation summary to explicitly satisfy the time-of-day vs service-type
requirement.

PRIORITY 2 - FR9 HTTP status filter

Add HTTP status code as a global filter in the drawer. Suggested values:

    All Status Codes, 200, 304, 404, 500

This is a small UI addition because status_code already exists in the dataset.

PRIORITY 3 - FR10 demo vs event separation

Add two explicit Sales KPI cards or a small funnel table:

    /scheduledemo.php requests
    /event.php requests
    Warm leads as % of total traffic

This will directly match the SRS wording.

PRIORITY 4 - FR6 choropleth and underperforming-region flag

Either add a proper Plotly choropleth or document why the current scatter-map
approach is used. Add a flag for countries below target_traffic_share or
target_potential_customer_share, labelled "Below expansion target."

PRIORITY 5 - FR2 ETL records processed/cleaned/dropped summary

Expose a small ETL summary in the Data & Export section:

    Loaded X rows
    Cleaned Y rows
    Dropped Z malformed rows

Even if Z is zero, displaying the summary closes the requirement.


--------------------------------------------------------------------------------
SECTION 8 - OVERALL VERDICT
--------------------------------------------------------------------------------

CyberNova is a strong Streamlit BI prototype that meets most of the SRS and
exceeds the baseline in several areas. The current modular version is stronger
than the older single-file implementation because it separates Sales, Marketing,
Executive, export, and generation concerns into different files.

The strongest areas are synthetic data generation, dashboard design, role-based
access, service and AI analytics, forecasting visuals, export capability, and
dataset scale. The measured local response time and 60,551-row dataset give good
evidence that the prototype is viable for local demonstration.

The remaining gaps are focused and fixable. Before final sign-off, the app
should add HTTP status filtering, explicit demo/event warm-lead split, a
descriptive statistics/correlation section, and a clearer underperforming-market
flag. Once those are added, the system would satisfy all twelve functional
requirements and all eight non-functional requirements either fully or with
clear documented justification.
