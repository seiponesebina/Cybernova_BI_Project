===============================================================================
1. Purpose of This Evaluation
===============================================================================

This document evaluates the CyberNova Product Sales Data Analysis System against
the Software Requirements Specification (SRS). The SRS defines a Streamlit-based
Business Intelligence dashboard that analyses synthetic IIS web server logs for
CyberNova Analytics Ltd, a Gaborone-based AI cybersecurity start-up.

The evaluation checks whether the delivered system satisfies the functional
requirements, non-functional requirements, project constraints, and business
objectives described in the SRS.

Overall verdict: FULLY / SUBSTANTIALLY MET

The current CyberNova BI system delivers the required synthetic IIS log analysis
solution and goes beyond the minimum SRS in several areas. The implementation
includes a reproducible synthetic data generator, enriched BI dataset, role-based
Streamlit dashboards, login-controlled stakeholder views, live replay feed,
geographic and service analytics, warm lead tracking, forecasting, exports, and
technical deployment readiness through GitHub and Streamlit.

The strongest areas are:

- Synthetic IIS W3C-style log generation at scale.
- Southern African market modelling with Botswana as the leading market.
- Role-specific Sales, Marketing, and Executive dashboards.
- AI Cyber Assistant flagship product tracking.
- Warm lead, event, demo, campaign, and conversion analysis.
- ETL transparency, validation evidence, and export support.
- Deployability using open-source Python and Streamlit.

The remaining limitations are controlled prototype limitations rather than
blocking failures. Live monitoring is implemented as a synthetic replay feed,
role authentication is suitable for demonstration but not production security,
and forecasting uses simple interpretable trend models rather than advanced
time-series modelling.

4. Functional Requirements Evaluation
===============================================================================

FR1 - Generate at least 1,000 rows of synthetic IIS W3C log data
Status: FULLY MET

The system generates 60,551 rows, far above the minimum 1,000-row requirement.
The raw CSV includes IIS-style fields including date, time, IP address, HTTP
method, URI, status code, user agent, bytes transferred, and response time.

The generator also models realistic CyberNova traffic, including Southern
African country distribution, Botswana as the leading market, service demand,
campaign windows, bot traffic, peak-hour behaviour, anomalies, and web errors.


FR2 - Ingest and preprocess logs, resolve countries, handle malformed records,
and report processed, cleaned, and dropped records
Status: FULLY MET

The application loads the enriched CSV as the primary source, parses timestamps,
casts boolean fields, sorts records chronologically, caches the dataframe, and
supports fallback loading from Excel in the reference implementation.

The generation pipeline creates validation and quality fields such as status
validity, anomaly flags, freshness indicators, and quality summaries. The
dashboard also includes ETL/context evidence explaining raw input, enrichment,
cleaning, and dashboard-ready output.

Country handling is implemented through configured/static country assignment,
which matches the SRS constraint that no paid live geolocation API is required.


FR3 - Perform EDA with distributions, country-by-service analysis, time series,
anomaly detection, and interpretive commentary
Status: FULLY MET

The dashboards provide exploratory analysis through KPIs, service rankings,
country summaries, campaign analysis, funnel views, time trends, anomaly
indicators, market maps, status-code health, and stakeholder-specific narrative
interpretation.

EDA is presented in business-facing form rather than as one isolated academic
notebook. The result still satisfies the SRS because distributions, bivariate
country/service views, time-based analysis, and anomaly review are available in
the role dashboards and data/export views.


FR4 - Segment traffic into four behavioural tiers and report volumes per country
Status: FULLY MET

The system implements the required four behavioural segments:

- High-intent.
- Product-curious.
- General browser.
- Bot.

The enriched dataset contains these segment labels, and the dashboards use them
for filters, summaries, and audience interpretation. Country and market views
allow segment behaviour to be inspected geographically.


FR5 - Simulate a real-time log feed and refresh dashboard metrics
Status: FULLY MET FOR PROTOTYPE SCOPE

The project includes data/live_stream_queue.csv and live-feed logic in the
Streamlit app. cybernovaapp.py releases synthetic live rows at configurable
speeds, updates timestamps to the current replay moment, and refreshes the view
using streamlit-autorefresh when available. app_html.py also documents the
earlier fragment-based replay concept.

Because the SRS states that real CyberNova server logs are unavailable, a
synthetic replay is an appropriate and compliant implementation. It should be
described as a simulated operational feed, not as a production server listener.


FR6 - Geographic demand analysis with ranked bar chart, choropleth, top five
markets, and underperforming-region flags
Status: FULLY MET

The system includes market-level ranking, map-based analysis, Southern African
country coverage, target status outputs, and below-target region flags. The
reference app includes Plotly choropleth logic and explicit target status labels
such as "Below expansion target."

The generated data supports the geographic requirement with ten regional
markets: Botswana, South Africa, Zambia, Namibia, Zimbabwe, Lesotho, Eswatini,
Mozambique, Malawi, and Tanzania.


FR7 - Rank service demand and isolate AI Cyber Assistant as a flagship KPI with
a growth trend indicator
Status: FULLY MET

Service demand is available through service_name and service_category fields and
is presented through rankings, KPIs, and trend views. AI Cyber Assistant is
treated as a flagship product through dedicated service fields, AI-interest
flags, AI conversion logic, and executive summaries.


FR8 - Three-tab Streamlit dashboard for Sales, Marketing, and Management
Status: FULLY MET

The system provides three stakeholder dashboard areas:

- Sales / CyberNova Pulse.
- Marketing / CyberNova Reach.
- Executive Management / CyberNova Horizon.

The current design goes beyond the minimum by adding multiple internal tabs for
overview, analytics, forecasting, and data/export workflows. Role-based access
limits each user to the dashboard relevant to their responsibility, while the
admin/lecturer role can access all dashboards.


FR9 - Sidebar filtering by date range, country, service type, and HTTP status
Status: FULLY MET

The system includes filtering by:

- Date range.
- Market / country.
- Service.
- Segment.
- Visit outcome / HTTP status class.
- HTTP status code in the current cybernovaapp.py filter state.
- Bot / filtered traffic inclusion.

The earlier evaluation marked HTTP filtering as a gap, but the current code
contains explicit status-code/status-class filtering support.


FR10 - Warm lead tracking for /scheduledemo.php and /event.php separately
Status: FULLY MET

The generated data includes warm lead indicators, demo request flags, event
signup flags, funnel stages, event types, and URI-level tracking. The dataset
contains 1,174 /scheduledemo.php records and 1,629 event-related records, so the
system can quantify demo and event interest separately.

Sales dashboards use these fields for lead prioritisation, funnel analysis,
country split, action queues, and export evidence.


FR11 - Provide at least five chart types with titles, axis labels, legends, and
annotations
Status: FULLY MET

The system uses more than five chart types, including:

- KPI cards.
- Bar charts.
- Line and area charts.
- Pie/donut charts.
- Choropleth/map visuals.
- Heatmaps.
- Funnel-style views.
- Tables.
- Forecast visuals.

Chart titles, contextual labels, legends, and interpretation text are present
throughout the dashboard views.


FR12 - Provide descriptive statistics by country, service, and time period, plus
correlation analysis of time-of-day versus service type
Status: SUBSTANTIALLY MET

The enriched dataset contains the required dimensions: country, service_name,
service_category, date, timestamp, hour, status class, segment, response time,
bytes transferred, and lead/conversion fields.

The dashboard presents totals, rankings, rates, trends, peak-hour indicators,
service demand, and market summaries. Time-of-day versus service demand is
supported through hour and service fields and visual heatmap-style analysis in
the dashboard/reference implementation.

For final academic presentation, this requirement is strongest when the Data &
Export or analytics tab is used to show:

- Mean, median, standard deviation, and totals by country.
- Mean, median, standard deviation, and totals by service.
- Daily/weekly/monthly time summaries.
- Hour-by-service matrix or heatmap.
- A short interpretation of peak service demand windows.

===============================================================================
8. Controlled Limitations
===============================================================================

1. Live feed is a replay simulation.
   This is acceptable for the project because real server logs are unavailable.
   It should not be presented as a production server ingestion service.

2. Authentication is prototype-level.
   Current role passwords are suitable for demonstration. Production deployment
   would need hashed credentials, secrets management, audit logs, and stronger
   session management.

3. Forecasting is planning-level.
   Forecast charts use simple interpretable trend models. This is suitable for
   BI explanation but not a statistically validated production forecast model.

4. Large generated files affect GitHub/Streamlit deployment.
   The enriched CSV and Excel outputs are large. Streamlit deployment should
   either include only required data files or regenerate/cache data during build
   depending on repository size and cloud limits.
