===============================================================================
1. Purpose of This Evaluation
===============================================================================

This document evaluates the CyberNova Product Sales Data Analysis System against
the Software Requirements Specification (SRS). The SRS defines a Streamlit-based
Business Intelligence dashboard that analyses synthetic IIS web server logs for
CyberNova Analytics Ltd, a Gaborone-based AI cybersecurity start-up.

The evaluation checks whether the delivered CyberNova system satisfies the
functional requirements, non-functional requirements, project constraints, and
business objectives described in the SRS.

Overall verdict: SUBSTANTIALLY MET, WITH A FEW EVIDENCE GAPS

The CyberNova system demonstrates the main required product: a synthetic IIS log
generation pipeline, a BI-ready dataset, role-specific Streamlit dashboards for
Sales, Marketing, and Management, geographic analysis, service demand analysis,
warm lead tracking, behavioural segmentation, and export/reporting support.

The strongest areas are synthetic data generation, dashboard design, role-based
analytics, service and AI Cyber Assistant tracking, scalability, and ethical use
of synthetic data. The main remaining weaknesses are the need for clearer ETL
quality reporting, explicit HTTP status filtering, stronger real-time feed
evidence, more visible descriptive statistics, and a clearly labelled
time-of-day versus service-type correlation output.


===============================================================================
2. System Summary
===============================================================================

CyberNova Analytics Ltd is represented as an AI cybersecurity start-up serving
SMEs, financial institutions, and government agencies across Southern Africa.
The system being evaluated is a Product Sales Data Analysis System designed to
measure the online effectiveness of CyberNova's software solutions using IIS web
server log analysis.

The implemented system includes:

- Synthetic IIS-style web log generation.
- Enriched BI dataset creation.
- Static IP-to-country resolution for Southern African markets.
- Behavioural segmentation of visitors.
- Sales, Marketing, and Executive/Management dashboard views.
- Interactive visual analytics using Streamlit and Plotly.
- Global filtering and role-focused dashboard layouts.
- Warm lead, service demand, campaign, geographic, and AI product analysis.
- Export/reporting functionality.
- Configuration-driven repeatable generation through config.yaml.

Key implementation files include:

- cybernovaapp.py: main Streamlit application.
- sales_views.py: Sales dashboard views.
- marketing_views.py: Marketing dashboard views.
- executive_views.py: Executive/Management dashboard views.
- generate_logs.py: synthetic IIS log generation and enrichment.
- validate_dataset.py: dataset validation checks.
- exports.py: PDF, CSV, Excel, and summary export helpers.
- live_feed.py: simulated live feed support.
- config.yaml: seed, country mix, services, campaigns, anomalies, and outputs.

3. Functional Requirements Evaluation
===============================================================================

FR1 - Generate at least 1,000 rows of synthetic IIS W3C log data
Status: FULLY MET

The system generates a dataset far above the minimum requirement. Existing
project evidence shows an enriched dataset of more than 60,000 records. The
dataset contains IIS-style fields such as date, time, IP address, method, URI,
HTTP status code, user agent, bytes transferred, and response timing.

The generator also models realistic behaviour including Southern African traffic
distribution, Botswana as the leading market, peak-hour clustering, repeat
visits, crawler/bot traffic, campaign effects, anomalies, and 404 errors.

Evaluation: The requirement is fully satisfied.


FR2 - Ingest and preprocess logs, resolve countries, handle malformed records,
and report processed, cleaned, and dropped records
Status: MOSTLY MET

The system loads generated log data into a clean BI-ready dataframe. Timestamp
fields are parsed, countries are assigned through static mapping, boolean flags
are normalised, and enriched analytical fields are prepared for dashboard use.
This matches the SRS constraint that no live geolocation API is required.

The main gap is presentation. The SRS specifically asks for records processed,
cleaned, and dropped to be reported. The project contains validation and loading
logic, but the dashboard should show a clear ETL quality panel such as:

- Records loaded.
- Records cleaned.
- Records dropped or rejected.
- Missing-field count.
- Malformed-record count.

Evaluation: The technical preprocessing is present, but the visible ETL report
should be strengthened.


FR3 - Perform EDA with distributions, country by service analysis, time series,
anomaly detection, and interpretive commentary
Status: MOSTLY MET

The dashboards provide exploratory analysis through KPIs, service rankings,
country summaries, campaign analysis, funnel views, time trends, risk/anomaly
indicators, and business-facing notes. These outputs support the Sales,
Marketing, and Management audiences described in the SRS.

The system also generates anomaly fields and uses them in dashboard sections.
However, for academic evaluation, the EDA section would be stronger if it were
explicitly labelled and structured around the SRS terms: univariate analysis,
bivariate country-service analysis, hourly/daily time series, and anomaly
detection.

Evaluation: The analysis is substantially implemented, but formal EDA labelling
would make the SRS match easier to prove.


FR4 - Segment traffic into four behavioural tiers and report volumes per country
Status: FULLY MET

The system implements the four required behavioural segments:

- High-intent visitors.
- Product-curious visitors.
- General browsers.
- Bots/crawlers.

The data model includes segment fields and the dashboard uses segmentation in
analytics and filtering. The project also contains validation evidence showing
that all four segment categories are generated.

Evaluation: The requirement is satisfied. A country-by-segment table would make
the evidence even clearer, but the core requirement is implemented.


FR5 - Simulate a real-time log feed and refresh dashboard metrics
Status: PARTIALLY MET

The project includes live feed support and Streamlit refresh behaviour for
live-looking dashboard sections. This is appropriate because the SRS states that
no real CyberNova server logs are available.

The limitation is that the live feed is a simulation rather than a true file
watcher connected to newly appended log rows. To fully satisfy the strict SRS
wording, the dashboard should prove that new log entries are detected through
polling or file-watch logic and that all affected metrics refresh from those
new rows.

Evaluation: The intent is met, but strict real-time evidence remains partial.


FR6 - Geographic demand analysis with ranked bar chart, choropleth, top five
markets, and underperforming-region flags
Status: MOSTLY MET

The system includes country-level market analysis, maps, country rankings, and
market status outputs. The dataset includes Southern African country coverage
and supports top-market analysis.

The main issue is the exact visual and decision wording. The SRS specifically
requires a choropleth and flags for underperforming regions against expansion
targets. If the current dashboard uses marker maps or static market cards, this
should either be changed to a true choropleth or documented as an equivalent
geographic visual. Underperforming markets should be explicitly labelled using
target traffic share or target potential customer share.

Evaluation: Geographic BI is implemented, but the choropleth and target-gap
evidence should be made more explicit.


FR7 - Rank service demand and isolate AI Cyber Assistant as a flagship KPI with
a growth trend indicator
Status: FULLY MET

The system clearly supports service demand analysis and treats the AI Cyber
Assistant as a flagship product. AI-related traffic is isolated through service
fields, AI page indicators, KPIs, trend visuals, and management-facing summaries.

Evaluation: The requirement is satisfied.


FR8 - Three-tab Streamlit dashboard for Sales, Marketing, and Management
Status: FULLY MET, WITH WORDING NOTE

The system provides three stakeholder dashboard areas:

- Sales.
- Marketing.
- Executive/Management.

Each role is supported with tailored analytics, which matches the audience
structure required by the SRS. The current implementation appears to go beyond
the SRS by using multiple tabs per role, including overview, analytics,
forecasting, and export-oriented sections.

Evaluation: The requirement is satisfied. If the SRS is interpreted strictly as
exactly three tabs, the SRS wording should be updated to say three role-based
dashboards rather than exactly three tabs.


FR9 - Sidebar filtering by date range, country, service type, and HTTP status
Status: PARTIALLY MET

The project includes global filtering controls for date, country/market,
service, segment, and related dashboard dimensions. Filtered data is passed into
role-specific views.

The main gap is HTTP status filtering. The SRS explicitly requires HTTP status
as a filter. If status_code is not visible as a global filter, it should be
added with options such as 200, 304, 404, and 500.

Evaluation: Filtering is mostly present, but HTTP status filtering must be made
explicit for full compliance.


FR10 - Warm lead tracking for /scheduledemo.php and /event.php separately
Status: MOSTLY MET

The generated data includes warm lead fields such as demo requests, event
signups, event type, funnel stage, and warm lead indicators. The Sales dashboard
contains conversion and lead-focused analysis.

The SRS specifically asks for /scheduledemo.php and /event.php to be quantified
separately, with count, country split, time trend, and rate as a percentage of
total traffic. The system supports this analysis, but the dashboard should make
the two routes visibly separate in KPI cards, a funnel table, or a dedicated
warm-lead section.

Evaluation: The data foundation is strong, but the exact reporting format should
be made more explicit.


FR11 - Provide at least five chart types with titles, axis labels, legends, and
annotations
Status: FULLY MET

The system uses more than five chart types, including bar charts, line/area
charts, pie/donut charts, scatter charts, maps, heatmaps, funnels, and forecast
visuals. These are suitable for the different stakeholder audiences.

Evaluation: The requirement is satisfied. A final visual review should still
confirm every major chart has clear titles, labels, legends, and annotations.


FR12 - Provide descriptive statistics by country, service, and time period, plus
correlation analysis of time-of-day versus service type
Status: PARTIALLY MET

The dataset contains the fields needed for this requirement, including country,
service, timestamp, hour, day, segment, and service-level fields. The dashboards
also show summary statistics, totals, rankings, and trends.

The gap is that the SRS asks for specific descriptive statistics: mean, median,
standard deviation, and totals per country, service, and time period. It also
asks for correlation analysis of time-of-day versus service type. These should
be clearly visible in the dashboard or in an export, preferably as:

- A descriptive statistics table.
- A country/service/time summary table.
- An hour by service heatmap.
- A short interpretation of peak engagement windows.

Evaluation: The data supports the requirement, but the visible evidence is
partial unless these outputs are clearly displayed.

4. Non-Functional Requirements Evaluation
===============================================================================

NFR1 - Dashboard loads within 3 seconds and filters respond within 1 second
Status: MOSTLY MET / NEEDS FINAL TIMING EVIDENCE

The system uses Streamlit caching and efficient dataframe loading patterns. Prior
project evidence shows the app can launch successfully and serve locally within
the required time range. However, the filter response target should be verified
through a timed interaction test before final submission.

Evaluation: Likely met locally, but final evidence should include measured load
and filter response times.


NFR2 - IIS W3C format, realistic Southern African distribution, and at least 30%
Botswana traffic
Status: FULLY MET

The generated data contains IIS-style request fields and Southern African market
coverage. Botswana is configured as the leading country and existing validation
evidence shows the Botswana share exceeds the 30% minimum.

Evaluation: The requirement is satisfied. If a strict W3C text log file is
required, the project should add W3C header lines or document that the CSV uses
IIS W3C-equivalent fields.


NFR3 - Fixed random seed for reproducibility
Status: FULLY MET

The generator is controlled by config.yaml, including a fixed random seed. This
supports repeatable dataset generation and monthly re-execution.

Evaluation: The requirement is satisfied.


NFR4 - Consistent UI, readable font size, and appropriate information density
Status: MOSTLY MET

The dashboard has a consistent CyberNova visual identity, role-specific layouts,
and suitable density differences between operational and executive audiences.
Sales and Marketing views contain more detail, while Management views focus on
KPIs and executive summaries.

The only caution is that any very small CSS text sizes should be checked against
the SRS requirement for readable fonts.

Evaluation: The requirement is substantially satisfied.


NFR5 - Modular codebase with separate generation, ETL, analysis, and UI
Status: MOSTLY MET

The codebase is modular across generator, validator, main app, role view modules,
export helpers, configuration, and live feed support. This is suitable for a
single-analyst academic project.

Some responsibilities still overlap, especially where generation, enrichment,
analytics, and dashboard presentation meet. However, the current structure is
much stronger than a single-file prototype and supports monthly re-execution.

Evaluation: The requirement is substantially satisfied.


NFR6 - Pipeline handles up to 50,000 log entries without restructuring
Status: FULLY MET

Project evidence shows a generated dataset above 50,000 rows. The dashboard and
validation scripts are able to work with this scale using pandas and Streamlit
caching.

Evaluation: The requirement is satisfied.


NFR7 - Deployable locally or through Streamlit Community Cloud from
requirements.txt, with no proprietary BI platform
Status: FULLY MET

The project is implemented in Python and Streamlit with open-source libraries.
requirements.txt exists, and the application can be run locally using Streamlit.
No proprietary BI platform is required.

Evaluation: The requirement is satisfied.


NFR8 - Synthetic data only, no real personal data, and professional
documentation
Status: FULLY MET

The SRS and project documentation clearly state that all data is synthetic. The
generator creates simulated web traffic and does not depend on real CyberNova
server logs or real personal data.

Evaluation: The requirement is satisfied.

The project generates a large synthetic dataset with realistic Southern African

anomalies.

2. Role-specific BI design
The system separates Sales, Marketing, and Management needs instead of showing

4. Good scalability evidence
The system handles more than 50,000 rows, which satisfies the SRS scalability

===============================================================================
8. Main Gaps and Recommended Improvements
===============================================================================

Priority 1 - Add explicit HTTP status filtering
The SRS requires filtering by HTTP status. Add status_code to the global filter
drawer or sidebar.

Priority 2 - Add a visible ETL quality summary
Show records processed, cleaned, dropped, malformed, and loaded. Even if dropped
records are zero, the dashboard should say so.

Priority 3 - Strengthen FR12 statistics and correlation evidence
Add mean, median, standard deviation, and totals by country/service/time period.
Add an hour by service heatmap or correlation-style summary.

Priority 4 - Separate /scheduledemo.php and /event.php warm lead reporting
Show separate counts, rates, country splits, and time trends for both routes.

Priority 5 - Make real-time refresh evidence clearer
If possible, connect live_feed.py or a polling file source so new rows visibly
update dashboard metrics. If not, label the feature as simulated live replay.

Priority 6 - Confirm choropleth and underperforming-region flags
Use a true choropleth or document the current map as the chosen geographic
visual. Add an explicit below-target market flag.

Priority 7 - Run final performance validation
Record dashboard first-load time and filter response time to prove NFR1.


===============================================================================
9. Final Conclusion
===============================================================================

The CyberNova BI system is a strong match for the SRS. It successfully delivers
the main required product: a synthetic IIS log based BI dashboard for evaluating
CyberNova's online product effectiveness across Sales, Marketing, and
Management audiences.

Most functional and non-functional requirements are fully or mostly met. The
system is especially strong in synthetic data generation, role-based dashboard
design, AI Cyber Assistant analysis, geographic market analysis, segmentation,
exports, reproducibility, scalability, and ethical handling of synthetic data.

The system should not be described as perfectly complete until the remaining
evidence gaps are closed. The most important final improvements are HTTP status
filtering, ETL quality reporting, explicit descriptive statistics, correlation
analysis, separate demo/event warm lead reporting, and stronger live feed
evidence.

Final rating: SUBSTANTIAL PASS

The system is suitable for demonstration and client sign-off, provided the
remaining minor compliance gaps are either fixed or clearly documented as
controlled limitations.
